2 · Data model: type vs. portfolio vs. tag vs. related

  • type is already a first-class field (peer-reviewed, discussion/working paper, chapter, thesis, policy brief, op-ed, file drawer…) — the first axis of the model. It renders as the badge on every card and drives the wall's type filter. No tags needed to duplicate it.
  • portfolio[] — which initiative streams an item appears in (1+; multi-membership = the visible bridge between portfolios).
  • tags[] — cross-cutting facets: turkey, mena, experiment (survey/conjoint experiments: ids 7, 8, 10, 11, 12), polarization, civil-war, identity-politics, migration, refugees, dev-cooperation, policy-advice (commissioned/advisory work: ids 34, 37, 39, 49, 50 — also surfaces the under-represented advisory record), earlier-work, personal.
  • related: — explicit item-to-item links with a short label, rendered as a "connections" line on each card (and as drawn edges in network-style designs). Schema and semantics: 
    - id: 19            # GovTech reality check (PB17)
  related:
    - {id: 12, rel: "policy-brief version of"}   # CIMER DP35
- id: 29
  related:
    - {id: 30, rel: "companion argument"}        # both tech-upgrade op-eds
- id: 26
  related:
    - {id: 13, rel: "features the PROSEC dataset"}
- id: 46
  related:
    - {id: 11, rel: "early version of the argument"}
    Links render on both cards automatically (12 shows "policy brief: GovTech reality check →"). Typical labels: policy-brief version of · distills · features dataset · companion argument · early version of · grew into · pre-registration of.

4 · Cross-linkage & academic↔policy blending (goal 1)

  • Multi-portfolio membership — an item in two portfolios appears in both streams and is the visible bridge (G&O bridges 02↔01; CIMER bridges 01↔04↔05; PROSEC bridges 03↔05).
  • related: links — see §2 for schema; both cards show the connection.
  • Anchor projects — publications/highlights/posts can carry project: demcor | depol-toolkit | datahub, tying outputs to the live initiatives.
  • One interleaved stream per portfolio — papers, briefs, op-eds, datasets, talks, posts sorted by date with type badges. No academic/non-academic separation; the wall's type filter serves the academic-only view.

8 · Constraints & risks

  • GitHub Pages = stock Jekyll only: no build plugins, no npm. Interactives = vanilla JS reading JSON emitted by Liquid from _data/.
  • Package installs denied in this environment — local preview depends on an existing jekyll/bundler; else static mockups + CI build on push.
  • 05 is wide by design (the trunk), but wall-residuals still never enter it.
  • DERTT browser ships with a clear data-version note; never imply more coverage than the release.
  • Website is an embedded repo — commits separate from the workspace repo; never git add from the parent.
